Here we'll describe how the ticketing system should work for front-end and back-end users. We will then provide basic and more advanced specifications.

Using the ticketing system

Users are identified with their email addresses. Anyone can create a ticketing service, the service "belongs" to the one who created it. Anyone with a link to a service can fill-in tickets.

Back-end users

First, back-end users will setup the tool, choosing what fields the front-end users will have to fill for each ticket and what categories will they chose from to file the ticket. The ticketing system will then configure a database accordingly and provide a unique URL for the ticketing system. This URL will be the entrypoint for anyone needing to fill a ticket for that project.

A second URL is given to back-end users, that points to another interface that allows them to query the data from the database. They can add information/comments related to the ticket and mark it as solved, have access to stats regarding the tickets.

Front-end users

Following the URL given, the front-end users can fill in a new ticket using the fields cofigured. They can also access / update an old ticket to see its status. The front-end user is notified anytime one of his ticket is updated.

Specification

Basics

Interfaces

One interface for the configuration of the service by the backend user. He has to be able to fill in his email address, select what fields he wants the user to fill and can give a list of categories in which the user will file the tickets.

One interface for the front-end users. The user can either fill in a new ticket, or load and update an old ticket with its id.

Database

When the back-end user sets the service up, a table is created in the database to store the content of the tickets that will be filled later.

Email notifications

Backend user is notified when the database is setup and receives a link to the service. Backend user is notified whenever a ticket is filled or updated.

Front-end user is notified when his ticket is created and received the unique ID of his ticket. Front end user is notified when one of his ticket is updated.
